记一次生产事故中遇到的js拦截问题 2016-10-24|技术总结前端技术|js-拦截

项目中突然遇到生产环境中无法访问具体客户端页面问题，问题的症状是偶尔无法返回页面内容出现空白页的情况。项目中同事想了各种办法最后终于通过手机截包工具查到了问题的结症。

记录一下这次问题中处理的思路和细节

  1. 页面端无法访问无法提供服务，首先先查询后端日志上在出现问题的时间段是否有调用异常，是否有节点无法响应的情况 -> 结果：无
  2. 为了复现问题，找到了与汇报问题相同的手机环境和网络环境，查询公网ip后进行定点ip服务日志查询，针对提供服务的接口看是否有异常响应（异常响应状态，异常响应报文大小） -> 结果：无
  3. 为了复现问题，切换网络环境，通过局域网代理截包排查网络层请求问题，4g, 3g, 2g, wifi -> 结果：无
  4. 为了复现问题，通过安装手机端截包工具，iOS下HttpWatch，界面清晰，免费 -> 多次刷新查询到异常注入js请求

最终结果查询到某运营商通过DNS拦截在页面上注入了非法js，导致页面调用异常。

不查不知道，其实这样的例子网上真是数不胜数。

还是那句话，全网https，虽然这也是一条非常漫长的路。

DB2迁移Mysql的历程 2016-10-22|技术总结数据库技术|DB2-Mysql

项目里最近说的比较多的话题就是DB2转移Mysql迁移，确实在关系型数据库里，DB2在多种层面上是王牌利剑，但是就目前互联网电商的趋势来说，DB2的重量级和互联网轻盈多变的思想格格不入。
然而相比而知的Mysql却成为了国内互联网应用的宠儿，虽然语法并不是完全符合当年IBM提出的标准，但是完整的功能和多变轻量级的特性却似乎更加符合互联网数据存储的需求。

  1. 插入(valueA, valueB, valueC)键值对。

对比来说，Mysql的语句比较DB2更加的简洁，但是实际上Mysql的语句是不符合SQL标准的，所以on duplicate key update这样的语法可能也只能在Mysql中遇到，所以要注意别的关系数据库中，还是要遵循DB2中提供的事例走。

另外Mysql中还存在一种语法，可以实现类似的功能，那就是replace操作，但是如果细心执行一下replace和insert into .. on duplicate key update会发现，实际上replace执行了两次操作，而后者只执行一次操作。

这是为什么呢？实际上replace操作故名思议是替换操作，所以在实际操作过程中，replace操作会首先删除符合索引查询的记录，接着将需要更新的记录插入表中，这在存在外键或者需要通过主键自增来定位数据的场景下是不可取的。反之，insert into … on duplicate key update则只会执行插入或者更新操作，不会影响到当前记录的外键或者主键数据，所以具体场景还是要根据业务的需求来。

Union, union all, expect, expect all, left right inner join 和分页

记得最开始学习sql的时候，总会接触到很多奇奇怪怪sql，包含着很多外键和联表查询的操作，当时觉得很高级，并且每次都能在各自的逻辑里面绕好久，也记得接触left, right和inner join之后，每次有多表场景都喜欢用上几句以作为显摆。

然而事实这些语句关键字的使用出现率真的在一点点下降，几家公司待下来，除了商用数据库，对于查询结果要求很高的准确率，不太要求性能查询速度；基本上目前的电商环境下，联表查询和外键的使用时越来越少。应和Mysql的流行，DB2，Oracle等商用大型数据库的关键词使用习惯也随之改变。

数据查询和慢sql的查询依然是相当重要的，结合查询顺序建立合适的索引，遵循二叉树查询的原则，这些原理还是不变的，sql语句用explain做优化依旧是每一位开发工程师，DBA，构架师的常用操作之一。

曾经做过一个创业项目的sql语句优化工作，其中包含了大量的联表查询操作，看起来满足了业务的需求，实际上在最终表现上及其之差，查询操作经常会出现timeout，并且大量占用程序和数据源的资源。相反之的，在重构的时候，通过分表操作和合理的sql索引查询，大大的优化了系统的性能。

更则，但目前趋势下，利用空间换时间，通过分表分库换取更为高速的访问数据，更加简单的查询逻辑应景而生。业务驱动技术，大概如此吧。
