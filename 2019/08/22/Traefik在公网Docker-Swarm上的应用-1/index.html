Traefik在公网Docker-Swarm上的应用-1
2019-08-22|技术心得|

一点点背景

前文的Dokku应用描写了在单机上类似Heroku的运维实现。但是经过实际生产应用后，发现Dokku会带来比较多的问题。比方说：

  • 应用集成不透明，对前后端分离项目不友好。
  • 数据库插件需要进行定制化配置。
  • 整体运维体验基于命令行，控制操作记录比较差。
  • 单机上的应用竞争资源的情况比较多。

整体来说，Dokku是基于Docker的一个很方便的实现，比较适合测试配置，但是到了真正生产的应用会比较局限。
基于集群跨网络的实现对于管理不同级别的应用会有比较好的支持。

由于手头的远程机器配置都比较低（也就是穷），想玩K8S的念头在看到K8S的基础配置要求后凉的不能再凉。经过查阅资料看到了Docker本身通过Docker Swarm对分布式的支持。

What is the Docker Swarm
查看定义c
Docker Swarm is a clustering and scheduling tool for Docker containers. With Swarm, IT administrators and developers can establish and manage a cluster of Docker nodes as a single virtual system.

可以看到Docker Swarm提供了Docker节点之间的协同工作基础，特别是在局域网中可以通过网络通信构成虚拟协同分布式系统，帮助作业。

通常手上的VPS或者远程机器都是暴露在公网上的机器，想要建立集群是需要通过公网网路进行集成的。所以在Docker Swarm的搭建教程上，笔者也踩了很多的坑，注意需要在网路设置上多加注意才好。

建立集群

公网集群需要注意的点是要模拟私有网路，所以在广播地址的时候，一定要使用公网暴露地址才能建立正常的overlay网络。

搭建反向代理Traefik

Traefik是一款支持自动配置的反向代理&负载均衡服务。通过配置文件，开发人员可以很快速的发布部署后端应用并不需要手动配置反向代理。如下图：

以下特点特别吸引人：

  • 逻辑内外网隔离，和Docker Overlay网络配合分发内外网流量
  • 多域名支持，子域名自动分配支持
  • 自动化CD/CI支持
  • 支持各种HTTPS配置

顺口提一句，Caddy这款HTTP2服务器也很不错，而且还自带自动HTTPS。

顿时觉得Nginx和Haproxy真的是不方便。

内部路由

Traefik很厉害的功能是通过配置子域名分配前端路由规则，并根据后端内部服务器比重和配置进行流量分发。

举个栗子，我们有一个spring-boot的web service服务，想要暴露在公网子域名下协同对外提供服务。

通过docker-compose服务编排，可以将服务发布到docker stack中作为单独的服务。这个例子假设traefik已经做了单独部署，需要将app stack添加到traefik的虚拟局域网路中。
